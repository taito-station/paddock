# SQLite → Postgres データ移行 runbook（#36）

既存の `data/paddock.db`（SQLite）に蓄積した実データ（約 30 万行）を、Postgres へ移す手順。
コードは既に Postgres 専用（`PADDOCK_DB_URL=postgres://...`）。**スキーマは sqlx のマイグレーションで
作成し、データだけ pgloader で流し込む**（pgloader にスキーマ推論させず、ベースライン
`deployments/db/migrations` の型を正とするため）。

## 前提

- Postgres が起動している（`docker compose -f deployments/compose.yaml up -d`）。
- 移行元の SQLite ファイル（例: 既存 clone の `data/paddock.db`）が手元にある。
- ツール:
  ```bash
  brew install pgloader        # SQLite→PG のデータ移送
  cargo install sqlx-cli --no-default-features --features postgres   # マイグレーション適用
  ```

以下、移行先を `postgres://paddock:paddock@localhost:5432/paddock`、移行元 SQLite を
`/abs/path/to/data/paddock.db` とする（自分のパスに読み替える）。

## 手順

### 1. 移行先 DB をベースラインで初期化

空の移行先にスキーマを作る（`_sqlx_migrations` も正しく記録される）。

```bash
export DATABASE_URL="postgres://paddock:paddock@localhost:5432/paddock"
# 作り直す場合: scripts/reset-db.sh で空に戻してから
sqlx migrate run --source deployments/db/migrations
```

### 2. pgloader でデータのみ投入

ロード定義ファイルを作る（`/tmp/paddock.load`）:

```
LOAD DATABASE
    FROM sqlite:///abs/path/to/data/paddock.db
    INTO postgresql://paddock:paddock@localhost:5432/paddock

WITH data only, truncate, disable triggers, prefetch rows = 10000

EXCLUDING TABLE NAMES MATCHING '_sqlx_migrations', 'sqlite_sequence'

ALTER SCHEMA 'main' RENAME TO 'public';
```

- `data only`: 既に sqlx で作ったテーブルへ INSERT する（pgloader にテーブルを作らせない）。
- `truncate`: 投入前に対象テーブルを空にする（再実行を冪等にする）。
- `disable triggers`: FK 制約を投入中だけ無効化し、テーブル投入順の問題を避ける。FK 由来の
  system trigger を止めるため **superuser 権限が必要**（テーブル owner では不足。compose の `paddock`
  は superuser なので可）。非 superuser 環境では `disable triggers` を外し、親テーブルから順に
  個別投入（`INCLUDING ONLY TABLE NAMES MATCHING` で 1 テーブルずつ）する。順序の例:
  **親**（`races` / `horses` / `predict_sessions` / `race_cards`）→ **子**（`results` / `horse_past_runs` /
  `predict_bets` / `predict_race_conditions` / `horse_entries` / `predictions`）→ **孫**（`prediction_horses` /
  `prediction_bets`）。`race_odds` は FK 無しなので順不同。
- `EXCLUDING ...`: 旧 SQLite の `_sqlx_migrations`（旧 50 マイグレーション）と `sqlite_sequence` は移送しない。

実行:

```bash
pgloader /tmp/paddock.load
```

### 3. IDENTITY シーケンスを補正

`results.result_id` 等の `GENERATED BY DEFAULT AS IDENTITY` 列は、明示値を INSERT しても
内部シーケンスが進まない。次回の自動採番が既存 id と衝突しないよう最大値+1 へ進める。

```bash
psql "$DATABASE_URL" <<'SQL'
SELECT setval(pg_get_serial_sequence('results','result_id'),       COALESCE((SELECT MAX(result_id)       FROM results),0)+1, false);
SELECT setval(pg_get_serial_sequence('horse_entries','entry_id'),  COALESCE((SELECT MAX(entry_id)        FROM horse_entries),0)+1, false);
SELECT setval(pg_get_serial_sequence('predict_bets','bet_id'),     COALESCE((SELECT MAX(bet_id)          FROM predict_bets),0)+1, false);
SELECT setval(pg_get_serial_sequence('predictions','prediction_id'),COALESCE((SELECT MAX(prediction_id)  FROM predictions),0)+1, false);
SQL
```

### 4. 件数で整合性チェック

移行元（SQLite）と移行先（PG）の主要テーブル件数を突き合わせる。**件数は移行時点の SQLite を
基準にする**（参考: 2026-06-18 時点で races≈4891 / results≈68395 / race_odds≈181475 /
fetch_history≈479）。

```bash
# 移行元（SQLite）
for t in races results race_odds fetch_history race_cards horse_entries predictions; do
  printf "%-16s %s\n" "$t" "$(sqlite3 /abs/path/to/data/paddock.db "SELECT COUNT(*) FROM $t;")"
done
# 移行先（PG）
for t in races results race_odds fetch_history race_cards horse_entries predictions; do
  printf "%-16s %s\n" "$t" "$(psql "$DATABASE_URL" -tAc "SELECT COUNT(*) FROM $t;")"
done
```

> `results.status`（`NOT NULL DEFAULT 'finished'`）は、競走除外馬が混ざっていた古い取り込みでは
> 一律 `finished` になっている可能性がある（PDF パース由来の既知の制約）。status を正確に取り直すには
> 当該 PDF を再 ingest する（同じ `race_id` の UPSERT で全フィールドが上書きされる）。

### 5. アプリで動作確認

```bash
cargo run -p analyze -- <サブコマンド>     # 実データで集計が出る
cargo run -p predict  -- --summary ...     # 予想セッションが読める
```

## トラブルシュート

- `disable triggers` 実行に権限エラー → 接続ユーザが **superuser** か確認（owner では不足。compose の `paddock` は superuser）。
- 一部テーブルだけ件数が合わない → 当該テーブルを pgloader の `INCLUDING ONLY TABLE NAMES MATCHING '<name>'` で個別再投入し、手順 3 のシーケンス補正を再実行。
- 文字化け → SQLite 側はテキストが UTF-8 である前提。`encoding` 指定は不要（pgloader 既定で UTF-8）。
