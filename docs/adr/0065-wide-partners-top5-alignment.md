# 0065. ワイド相手頭数を top5 に統一（doc↔実装の乖離解消 — top3/top5 は有意差なし）

## ステータス

採用（accept）。`build_portfolio` は無変更（既に全券種 top5）。CLAUDE.md の買い方ルールを「ワイドは相手 top3」から「3 券種とも top5」へ修正し、doc を実装＝実測結果に一致させた。

## コンテキスト

CLAUDE.md 買い方ルールは「**ワイドは相手 top3 / 馬連・3連複は相手 top5**」と定めていた。一方、本番の
`build_portfolio`（`src/domain/src/portfolio/mod.rs`）は単一の `PortfolioConfig.partners`(=5) を全券種へ適用し、
**ワイドも相手 top5** で組んでいた。券種別の相手頭数フィールドは実装に存在せず、**doc とコードが乖離**していた。

「ワイド top3」の出所は ADR 0030 の混戦バックテスト（`konsen_backtest.py`）の baseline 実装が偶々 `wp=parts[:3]`
だっただけで、production の買い方構造（軸流し・券種別予算配分）で **ワイド top3 vs top5 を直接比較した根拠は
リポジトリ内に存在しなかった**。つまり乖離しているのは build_portfolio 側だが、それが検証済み構成から外れて
いるのか、doc の top3 が過剰な思い込みなのかが未確定だった（#347）。

安易にどちらかへ寄せず、**先に計測して決める（measurement-first）**方針を採った。

## 検証方法

- **ハーネス**: `scripts/predict-check/strategy_eval.py`。買い方（軸選定・券種別予算 `budget*w//total_w//100*100`・
  点数・1点¥100 丸め distribute・組番形式）は `build_portfolio` と等価であることを確認済み → ROI を production
  挙動の忠実な代理として使える。本 ADR 用に券種別相手頭数フラグ `--wide-partners`（ワイドだけ別 K・既定は
  `--partners` に追従）と、多日集計用の `--json` 出力を追加した。
- **リーク中立性**: ワイド top3 vs top5 は**予測確率不変・相手の絞りのみ変化**。preds はオッズブレンド由来の
  リーク（`analyze predict` は all-time 統計＝未来込み。単勝 ROI が 189〜594% と非現実的なのはこのため）を含むが、
  両条件に等しく効くため**相対比較（どちらが上か）はリーク中立**。ROI 絶対値は信用せず、差分と符号の安定性で判断。
- **突合キー**: strategy_eval は `(venue, race_num)` で突合し日付を含まないため、複数開催日を 1 run に混ぜると
  同一 venue+race_num が衝突する。**開催日ごとに run して合算**した（`agg347.py` / `perday347.py`）。
- **データ**: `gen_win_backtest_data.sh`（production 既定 α=0.2・m=10）で生成。窓は 2 段階。

## 結果

### 窓 A（71R / 4 開催日 2026-05-30〜06-14）

| 戦略 | wide=5 | wide=3 | Δ(3−5) |
|---|---|---|---|
| 本命軸 馬連+ワイド流し | 114.0% | 118.0% | **+4.0pt**（top3 有利） |
| 本命軸 馬連+ワイド+三連複流し | 93.4% | 96.5% | +3.1pt |

top3 が僅かに上に見えたが、この窓は単勝 ROI 594% の**幸運な小標本**だった。事前合意「差が微妙なら窓を
広げる」に従い N を拡大。

### 窓 B（262R 実測 / 290R 延べ / 12 開催日 2026-03-15〜07-05）

| 戦略 | wide=5 | wide=3 | Δ(3−5) |
|---|---|---|---|
| 本命軸 馬連+ワイド流し | 82.6% | 78.8% | **−3.9pt**（top5 有利） |
| 本命軸 馬連+ワイド+三連複流し | 75.4% | 72.5% | −2.9pt |

**N を増やすと符号が反転**した（A の top3 優位はノイズ）。さらに日別で見ると:

- **top5 ≥ top3 は 6/12 日、top3 > top5 は 6/12 日＝完全に五分**。
- 日別 Δ(3−5): +21 +4 −18 +3 +4 +8 +10 −14 **−32** −6 −4 −1（符号バラバラ）。
- B 窓集計の top5 優位（−3.9pt）は実質**単一の外れ日 2026-06-27（Δ−32pt / 34R）が牽引**しており、
  これを除けばほぼゼロ。

## 決定

**ワイド top3 vs top5 に安定した有意差は無い**（ノイズ支配・符号は標本と単一外れ日に依存）。有意差が無い以上、
**最小変更**を選ぶ:

- `build_portfolio` は**無変更**（既に全券種 top5）。ビュー表示（軸込み 6 頭のワイド）も現状維持。
- CLAUDE.md を「**3 券種とも相手 top5**」へ修正し、doc を実装＝実測に一致させる。「ワイド top3」という
  未検証の縛りは撤去。
- 買い方の広さは **top5 が上限**（ADR 0030「相手を top5 より広げると悪化」は不変）。ワイドを top5 にするのは
  上限に合わせるだけで、これと矛盾しない。むしろ CLAUDE.md「印を打った馬は必ず買い目に絡める」「保険構造」
  とも整合する（ワイドも印馬を広く拾える）。

なお AI の買い目提案は相手 top5 に絞るが、レース盤ビューは軸込み 6 頭を提示し、**最終判断（張る/見送り/増額）は
人間**に委ねる decision-support（ADR 0055/0060 の軸ロック運用と整合）。

## 影響

計測中に **2 つ目の買い目エンジン `live_ev.py`（predict-watch / refresh_ev.sh のライブ伝票）も
ワイド top3（`wp = parts[:3]`）** だったことが判明した。build_portfolio（top5）と食い違っており、
top5 へ統一しないと #347 が潰す乖離が別エンジンに残るため、これも top5 に揃える。

- `CLAUDE.md`: 買い方ルール「相手の広さ」と「軸ロックとズレ増額」の 2 箇所を top5 統一に修正。
- `docs/specifications/betting-rule-history.md`: 本計測を履歴（⑩）に追記（top3 の再提案防止）。
- `scripts/predict-check/strategy_eval.py`: 券種別相手頭数フラグ `--wide-partners`（既定=`--partners` 追従＝
  後方互換）と多日集計用 `--json` 出力を追加。既存挙動（`--wide-partners` 未指定）は不変。
- `scripts/predict-check/live_ev.py`: ライブ買い目のワイドを `wp = parts[:3]`（top3）→ `wp = parts`（top5）
  に変更。ライブ伝票のワイドが軸込み 4 頭→6 頭になる。`docs/specifications/live-ev-buy-view.md` の
  「wide top3」注記も top5 に更新。`test_live_ev.py` はワイド点数を固定していないため不変で通る。
- 本番 Rust コード（`build_portfolio` / `PortfolioConfig`）・統合テストは不変（既に top5）。

## 再検証の条件

現データ（12 開催日・リーク有り preds の相対比較）では有意差なしだが、絶対 ROI はリークで信用できない。
リーク無しの as-of 確率（`analyze backtest --dump-features`）＋実配当が数百 R 規模に蓄積し、かつ**単一外れ日に
依存しない安定した符号**が出た時点で再検証する（#248 の snapshot 蓄積が母集団要件を満たしたら）。
