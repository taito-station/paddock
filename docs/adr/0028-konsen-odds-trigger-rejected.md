# ADR 0028: 混戦判定にオッズ条件を併用しない（バックテストで棄却）(Issue #180)

## ステータス
承認済み

## コンテキスト
予想の買い方には「混戦」を自動判定して印馬の3連複ボックスを別枠で1枚足す仕組みがある
（買い方ルールはローカルメモリ `feedback_betting_staking` と予想時の手計算で運用。本体コードの
`build_portfolio` には未取り込み）。現行の混戦判定は **◎の model 勝率の 0.70 倍以上の馬が ◎含め
4頭以上**（以下 `band≥4`）で発動する。

2026-05-30 京都11R 葵S（G3）で、◎が1番人気でも単勝 3.8 倍と支持が割れた「実質オープン戦」を
混戦と判定できず取りこぼした（band=2 のため非発動）。そこで **「◎単勝オッズが割れている（弱い
1番人気）なら band が薄くても混戦扱いにする」** 案（#180）が出た。issue では「閾値はバックテストで
決める／過学習に注意」を前提条件にしていた。

そこで再実行可能なバックテストハーネス `scripts/predict-check/konsen_backtest.py` を作り、確定買い方
（¥5,000・ワイド/馬連/3連複を model 確率で重み付け配分・混戦は3連複ボックス追加）を全レースへ機械適用し、
混戦の発動条件を切り替えて回収率を比較した。

測定窓: **2026-05-30 / 05-31 / 06-13 / 06-14 の 71R**（DB に確率表・単勝オッズが揃う全レース）。
回収率の分母は **実際に賭けた額（実消化額）** を用い、ボックスが組成不能で余る予算を損失計上して特定
variant を不当に不利化しないようにした。

## 決定
1. **混戦判定にオッズ条件は併用しない。現行 `band≥4` を維持する。** どの閾値でも回収が悪化したため。
2. 検証ハーネス `konsen_backtest.py` は再実行可能な形で残す（将来 days を足して再評価できる）。
3. 判定式（買い方の発動条件）は **単発の成功例で恒久ルール化しない**。スコア/収支挙動を変える改善は
   バックテストで改善が出てから採用する（[[feedback_measurement_ordering]] と整合）。

### バックテストによる検証（71R, 実消化額ベース）

| variant | 回収率 | 混戦数 | 混戦時払戻 |
|---|---|---|---|
| box-off（混戦を一切発動しない純◎軸） | **84.0%** | 0 | 0 |
| baseline（band≥4 ボックス, 現行） | 83.9% | 3 | 1 |
| +odds≥3.0 / band≥3 | 81.5% | 7 | 3 |
| +odds≥3.0 / band≥2 | 80.8% | 20 | 10 |
| +odds≥3.5 / band≥2 | 79.5% | 13 | 8 |
| +odds≥4.0 / band≥2,3 | 83.9% | 3 | 1 |

**「◎単勝≥X倍で混戦扱い」はどの閾値（3.0/3.5/4.0 × band≥2/3）でも回収を悪化**させた（84% → 79.5〜81.5%）。
band≥2 まで緩めると 13〜20R を混戦化するが、回収はむしろ下がる。

## 理由
- 動機だった葵S（◎3.8倍）は band=2 で、3連複ボックスが **組成不能（3頭未満で C(band,3)=0）**。混戦化しても
  予算が余るだけで払戻 ¥0。オッズ条件は「拾いたかったレース」をそもそも拾えない。
- band≥2 まで緩めて混戦化する 13〜20R の多くは、model 下位の人気薄が突入する波乱で、ボックスでも当たらない。
  ◎軸に厚く張る（box-off/baseline）方が回収が良い。
- そもそも現行 `band≥4` ボックスの寄与も中立（box-off 84.0% と baseline 83.9% は誤差 1R）。
  白百合S（2026-05-31）でボックスが ◎着外でも当たった好例は、小標本のノイズ寄りだった。
- レビュー指摘を受け回収率の分母を「実消化額」に修正（賭けられない予算を損失計上しない公平な会計）した
  あとも結論は不変。むしろ odds variant の数値は補正後に上がってなお baseline 未満で、結論はより頑健。

## 影響
- 買い方ルールに変更なし（オッズ条件は入れない）。本体コード（`build_portfolio`）の挙動も変更なし。
- 検証基盤 `konsen_backtest.py` が `scripts/predict-check/` に追加され、混戦・配分まわりの将来の閾値検討を
  再実行可能にした。
- ローカルメモリ `feedback_betting_staking` に「オッズ条件は本バックテストで棄却・単発成功で恒久化しない」を記録。

## スコープと限界（過大結論を避けるための明記）
- **71R・4 開催日の小標本**。混戦自体が稀（baseline で 3R）で、効果量の小さい比較はサンプル依存。ただし本決定は
  「複雑化（オッズ条件）を入れない」という保守側の結論で、エンハンス側に改善の挙証責任があるところ全閾値で
  改善が出なかったため、追加複雑化を見送る根拠としては十分。
- 本 ADR は「オッズ条件を入れない」までを決める。**band≥4 ボックスそのものの是非**（box-off がわずかに上回る点）は
  小標本のため確定せず、現行維持として別途より大きなサンプルで再評価しうる。
- 買い方ロジックは現状コード化されておらず（手計算 + harness）、本 ADR は本体実装の変更を伴わない方針決定である。
