# ブラウザテストケース: レース結果の同日取り込みと UI 自動反映（#381）

対象: SPA `web/src/routes/RaceList.tsx`（`/races?date=YYYY-MM-DD`）・`SessionSummary.tsx`・`RaceBoard.tsx`。設計書 [race-result-ingestion.md](../../docs/specifications/race-result-ingestion.md) / ADR [0068](../../docs/adr/0068-race-result-ingestion-ui-reflection.md)。

検証環境は Playwright MCP 不在のため **headless Chrome + puppeteer-core** で代替する（`reference_browser_test_fallback`）。web 検証スタックは `reference_web_verify_stack`（colima → api-server をソースから起動 → vite `PADDOCK_API_TARGET`）。DB を汚さない再現は CDP レスポンス差し替えを用いる。

> これらは **実装 PR で実施する受け入れケース**。設計 PR 時点では設計に対する事前定義（実装は未着手）。

### TC-01: 発走済みレースが結果確定で「⚫終」に変わる（着順取り込みが根拠）
| 項目 | 内容 |
|------|------|
| 前提 | 当日開催。あるレースが `post_time` を過ぎ、`results:refresh` により着順が `results` に取り込まれている（`result_confirmed=true`）。別レースは `post_time` 未到達（未確定） |
| 画面 | `/races?date=YYYY-MM-DD`（ライブモード） |
| 操作 | 画面を開き、状態フィルタ「全部/未発走/終了」を切り替える |
| 期待結果 | `result_confirmed=true` の行に ⚫終 が付き「終了」フィルタに入る。未到達レースは「未発走」側。⚫終 の根拠が `post_time` の相対推定ではなく結果確定であること（未確定だが post_time 経過のレースは ⚫終 にならない） |
| 確認ポイント | `post_time` 経過かつ未確定のレースを ⚫終 と誤判定しない。post_time が null のレースは終了と断定しない（#391 踏襲） |

### TC-02: 発走済み行に着順・的中○/✗・払戻額が出る
| 項目 | 内容 |
|------|------|
| 前提 | 確定済みレースで、セッションに当該レースの買い目（的中1点・不的中複数）がある。着順が `finish_order` に入っている |
| 画面 | `/races?date=YYYY-MM-DD` |
| 操作 | 確定行を確認する |
| 期待結果 | 着順（上位）が表示される。賭けたレースは的中○/✗と払戻額（session `bets[].payout` 由来）が出る。的中は payout>0、不的中は 0 表示 |
| 確認ポイント | 賭けていない確定レースは着順は出るが的中/払戻欄は空（session に bet が無い）。払戻額が捏造されない。**軸が 4 着以下（`finish_order` 上位 3 に不在）で不的中のケースでも、的中/払戻判定は session `bets[].payout=0` から正しく ✗ 表示される**（着順表示 top3 と的中集計は別ソース＝突合が着順 top3 に依存しない） |

### TC-03: 未確定が残る間だけポーリングし、全確定で停止する
| 項目 | 内容 |
|------|------|
| 前提 | 当日、`post_time` 経過だが未確定のレースが 1 つ以上残る状態から開始。途中で全レースが確定に遷移する（CDP でレスポンスを段階差し替え） |
| 画面 | `/races?date=YYYY-MM-DD`（または収支サマリ） |
| 操作 | Network を監視しつつ 60〜120 秒放置。全確定後さらに放置 |
| 期待結果 | 未確定が残る間は `POST /api/results/{date}:refresh` が 30–60 秒間隔で発火。全レース確定後はポーリングが**止まる**（以降 `results:refresh` が飛ばない） |
| 確認ポイント | web 側（CDP/Network 観測）ではポーリングの発火/停止のみ検証する。**確定済みレースへ netkeiba 由来の再取得を誘発しないこと（サーバ側 gating・debounce）は CDP レスポンス差し替えでは検証できない**ため、api-server のログ観測、または `ResultsInteractor` の gating を対象とした統合テストで別途担保する（本ケースの範囲外として明記） |

### TC-04: 収支サマリが結果確定を検知して自動精算・自動反映する
| 項目 | 内容 |
|------|------|
| 前提 | 進行中セッション。未精算の確定レースがある |
| 画面 | 収支サマリ（`SessionSummary.tsx`） |
| 操作 | 画面を開いて放置（手動ボタンは押さない） |
| 期待結果 | ポーリングで自動精算が走り、残高・総払戻・回収率・settled/pending 数が手動操作なしで更新される。全レース確定で `completed` 表示になる |
| 確認ポイント | 冪等：ポーリングが複数回走っても二重加算しない（払戻・残高が発火回数で変動しない） |

### TC-05: 手動「精算」ボタンがフォールバックとして機能する
| 項目 | 内容 |
|------|------|
| 前提 | ポーリングを無効化した状態（もしくは過去日表示）で未精算の確定レースがある |
| 画面 | 収支サマリ |
| 操作 | 「精算」ボタンを押す |
| 期待結果 | `results:refresh`（委譲エイリアス含む）が **`force=true`** で発火し、着順取り込み＋精算が走って収支が反映される |
| 確認ポイント | 自動ポーリング（`force=false`）と手動ボタン（`force=true`）で確定済みレースの結果は一致する（同じ冪等フロー） |

### TC-05b: 手動ボタンが post_time 欠損レースを救済する（force=true の gating 緩和）
| 項目 | 内容 |
|------|------|
| 前提 | `post_time` 未取得（`race_cards` に post_time 無し・#391 で自動対象外）だが結果は既に出ている未確定レースがある |
| 画面 | 収支サマリ / ライブ一覧 |
| 操作 | 自動ポーリングを一定時間観測後、手動「精算」ボタンを押す |
| 期待結果 | 自動ポーリング（`force=false`）では当該レースは対象外で確定しない。手動ボタン（`force=true`）で post_time gating が緩和され、当該レースの結果取得＋確定＋精算が走る |
| 確認ポイント | 自動と手動の gating 差（post_time 欠損レースは force=true のみが拾う）が設計通り。確定済みレースは force=true でも再取得しない |

### TC-06: 前日・過去日はポーリングしない（静的表示）
| 項目 | 内容 |
|------|------|
| 前提 | 過去日（当日でない）を表示。結果は確定済み |
| 画面 | `/races?date=YYYY-MM-DD`（過去日） |
| 操作 | 画面を開いて放置 |
| 期待結果 | `results:refresh` が自動発火しない（静的に永続化済みデータを表示）。着順・的中/払戻・⚫終 は表示される |
| 確認ポイント | SPA 鮮度方針（再現性重視・過去日は自動更新しない）を崩さない |

### TC-07: 全馬取消・開催中止レースの扱い
| 項目 | 内容 |
|------|------|
| 前提 | 全額返還レース（全馬取消/中止、#131）。(a) netkeiba 結果ページに成績表（全馬取消行）が出るケースと、(b) 成績表自体が生成されないケースの両方を用意。当該レースに買い目がある |
| 画面 | `/races?date=YYYY-MM-DD` / 収支サマリ |
| 操作 | 確定後の表示を確認 |
| 期待結果 | (a) `fully_refunded` を検知して「確定（返還）」扱い・⚫終 が付き、買い目は全額返還（払戻=賭金）で収支反映。(b) 成績表が出ないケースは `post_time` からの経過タイムアウトで確定扱いにする（または手動フォールバックで確定）。着順 NULL でも最終的に確定へ収束する |
| 確認ポイント | (b) で成績表が生成されない間、**タイムアウト到達までは pending だが、到達後は確定して延々ポーリングし続けない**。返還が的中扱いで過大集計されない |

### TC-08: 1 レース盤に着順が反映される
| 項目 | 内容 |
|------|------|
| 前提 | 確定済みレースの盤を開く |
| 画面 | `RaceBoard.tsx`（`/races/{race_id}/board`） |
| 操作 | 盤を開く |
| 期待結果 | 各馬に着順（`finishing_position`）が表示され、盤が結果確定状態（`result_confirmed=true`）を反映する |
| 確認ポイント | 未確定レースの盤では着順欄が空で、予想（確率・買い目）表示は従来どおり保たれる |
